Техническое задание (ТЗ) на разработку системы регистрации участников конкурса

1. Введение

Цель проекта – создать backend-систему для конкурса, которая позволит:

  • Регистрировать участников через форму записи.
  • Получать список всех участников.
  • Объявлять результаты для всех участников.
  • Устанавливать статус победителя.

Система должна быть реализована на Java с использованием современных фреймворков и принципов разработки. Если вдруг возникнет вопрос «Кто победил?», API ответит без лишних приукрас – всё честно, прозрачно и с лёгкой долей юмора!

2. Функциональные требования

2.1 Регистрация участников

  • Эндпоинт: POST /participants
  • Описание: Принимает данные участника в формате JSON и сохраняет их в базу данных.
  • Поля запроса (пример):
    • fullName (строка, обязательное) – ФИО участника.
    • email (строка, обязательное) – Электронная почта.
    • phone (строка, опциональное) – Контактный телефон.
    • additionalInfo (строка, опциональное) – Дополнительные сведения (например, возраст, город и т.д.).
  • Валидация:
    • Обязательные поля не должны быть пустыми.
    • Формат email должен соответствовать стандарту.
  • Ответ:
    • Код 201 (Created) и объект участника с присвоенным идентификатором.
    • В случае ошибки – соответствующий HTTP статус (например, 400 для неверного запроса).

2.2 Получение списка участников

  • Эндпоинт: GET /participants
  • Описание: Возвращает список всех зарегистрированных участников.
  • Функционал:
    • Возможность пагинации (например, параметры page и size).
    • Фильтрация по статусу или другим параметрам (при необходимости).
  • Ответ:
    • Код 200 (OK) и массив объектов участников.

2.3 Объявление результатов

  • Эндпоинт: GET /participants/results
  • Описание: Возвращает информацию о результатах конкурса для каждого участника.
  • Логика формирования результата:
    • Результат может быть рассчитан заранее (например, по определённым критериям) или обновляться администратором.
    • В объекте участника может присутствовать поле result (например, строка с оценкой или статусом: «Победитель», «Участник» и т.д.).
  • Ответ:
    • Код 200 (OK) и массив объектов с результатами.

2.4 Установка победителя

  • Эндпоинт: PUT /participants/{id}/winner
  • Описание: Позволяет установить для конкретного участника статус победителя. Если требуется, можно обеспечить логику, ограничивающую установку только одного победителя, либо автоматом сбрасывать статус у предыдущего победителя.
    Важно: данный эндпоинт должен быть защищён – статус победителя изменяет ход конкурса, а у каждого победителя, как известно, должен быть свой поклонник (ну или, по крайней мере, группа администраторов).
  • Параметры запроса:
    • id (в URL, обязательное) – Идентификатор участника, для которого устанавливается статус победителя.
    • Опционально: в теле запроса можно передавать дополнительные параметры, если потребуется расширенная логика (например, комментарий администратора).
  • Логика обновления:
    • Обновление поля участника (например, result) с присвоением значения «Победитель».
    • Если в системе разрешён только один победитель, предусмотреть проверку и, при необходимости, сброс предыдущего статуса.
  • Ответ:
    • Код 200 (OK) и объект участника с обновлённым статусом.
    • Если участник с указанным идентификатором не найден – код 404.
    • При ошибках валидации – код 400.

3. Нефункциональные требования

3.1 Технологический стек

  • Язык: Java (рекомендуемая версия – 11 или выше).
  • Фреймворк: Spring Boot.
  • База данных: H2 для разработки/тестирования; PostgreSQL или MySQL для production.
  • ORM: JPA/Hibernate.
  • Сборка: Maven или Gradle.
  • Документация API: Swagger/OpenAPI (для удобного тестирования и демонстрации эндпоинтов).

3.2 Качество кода и тестирование

  • Написание unit-тестов с использованием JUnit и Mockito.
  • Логирование ошибок и информационных сообщений (например, с использованием SLF4J).
  • Обработка исключений: глобальный обработчик ошибок (ControllerAdvice).

3.3 Безопасность

  • Валидация входных данных.
  • Ограничение доступа к административным функциям – эндпоинт установки победителя должен быть доступен только пользователям с соответствующими правами.
  • Использование HTTPS в production-среде.

3.4 Производительность

  • Обеспечение быстрого отклика API.
  • Пагинация для GET-запросов, если количество участников велико.
  • Возможное кэширование для часто запрашиваемых данных (если требуется).

4. Архитектура и структура проекта

4.1 Структура слоёв

  • Контроллеры (Controllers): Принимают HTTP-запросы, вызывают сервисный слой и возвращают ответы.
  • Сервисный слой (Services): Содержит бизнес-логику для регистрации, получения участников, установки победителя и расчёта/объявления результатов.
  • Репозитории (Repositories): Работа с базой данных с использованием Spring Data JPA.
  • Модели (Entities/DTOs): Определяют структуру данных участников и результатов.

4.2 REST API

  • Эндпоинты должны соответствовать RESTful архитектуре.
  • Использовать стандартные HTTP методы (GET, POST, PUT/PATCH).

  3. Тестирование:

  4. Документация:

6. Дополнительные замечания

  • Разработка должна вестись с использованием системы контроля версий (например, Git).
  • Рекомендуется использование CI/CD для автоматизированного тестирования и сборки.
  • При возникновении спорных вопросов решение должно приниматься оперативно и практично – времени на излишние обсуждения не тратим, а победителя выбираем честно!

Таким образом, добавленный эндпоинт позволит оперативно установить статус победителя конкурса, что особенно актуально в моменты, когда нужно быстро определить, кто заслужил звание чемпиона.
